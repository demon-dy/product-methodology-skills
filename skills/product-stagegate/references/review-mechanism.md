# 阶段评审机制 + 与 OKR/敏捷的关系

方法论定义"什么算通关"，评审机制定义"谁来判定、怎么开会"。没有它，方法论容易被强势的"感觉判断"绕过。

## 阶段评审会

每次阶段跃迁开一次评审会。出席与职责：

| 角色 | 职责 |
|---|---|
| 产品负责人 | 主持，提交评审材料，陈述通关依据 |
| 技术负责人 | 确认数据采集口径准确、技术层面无遗漏 |
| 业务负责人/CEO | 最终拍板人（跃迁要投资源，需决策层确认） |
| 数据/运营（如有） | 交叉核验数据，提出质疑 |

**核心原则**：任何人可质疑，但拍板权归业务负责人；质疑必须基于数据，不接受"感觉还没到"。

## 各阶段必备评审材料

| 跃迁 | 必须提交 |
|---|---|
| Demo→MVP | 访谈记录（含被访者画像）、痛点确认率统计、反例清单 |
| MVP→PMF | 激活率数据、激活路径漏斗图、典型用户使用录屏（可选） |
| PMF→GTM | 留存曲线（Day7/14/30）、Sean Ellis 原始结果、用户分群分析 |
| GTM→MTU | 各渠道 CAC/LTV 对比表、回收期测算、单位经济健康度总结 |
| MTU 持续运营 | K-factor 趋势、自然增长占比、飞轮机制效果复盘 |

## 卡住时怎么办

- **数据不足**：不评审，补数据后重约，原则上不超 2 周
- **结论有分歧**：产品负责人与业务负责人 1:1 对齐，以业务负责人判断为准
- **连续两次未通关**：触发"方向复盘会"，重审核心假设（见 `self-check.md` 止损规则）

## 与 OKR / 敏捷的关系

- **早期（Demo/MVP）**：方法论优先级高于 OKR 数字目标。为完成季度 OKR 跳过验证，是用速度换方向正确性。KR 应设为"完成 X 次访谈 + 得出明确通关/不通关结论"，而非"DAU 达到 X"。
- **PMF 阶段**：不适合强推 Scrum 功能交付节奏。核心工作是等留存数据、做访谈、分析。建议以 2 周为一个"验证循环"，功能开发服务于"改善留存"这一单一目标。
- **GTM 及之后**：方向已定，OKR 管目标、Scrum 管交付节奏，两者正常使用。
