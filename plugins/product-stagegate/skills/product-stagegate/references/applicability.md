# 适用性

生搬硬套框架是常见失败模式。先判断该不该用，再用。

## 适用（同时满足效果最好）

1. 面向真实用户的新产品（C 端 / 小 B SaaS）——核心逻辑是"用最小代价验证市场是否存在"。
2. 早期 0→1 / 1→10，资源有限、方向不确定。
3. 输不起一次"all-in 赌错方向"——分阶段=把赌注分散到多次小赌。
4. 团队接受"数据 + 假设 + 证伪"的工作方式（文化不配合则摩擦大，先统一认知）。
5. 产品迭代空间较大（形态高度固定的定制项目价值打折）。

## 不适用（命中任一就换思路，别硬套）

| 场景 | 为什么不适用 | 改用什么 |
|---|---|---|
| ToG / 大客户强合同定制 | 合同在开工前就签了，问题不是"市场要不要"而是"按合同交付" | 项目管理（范围/工期/验收对齐）；AARRR 换成合同验收条款 |
| 监管/合规驱动功能 | 不是用户要不要，是法规必须有（实名认证、安全报告、风险提示） | 合规检查清单 + 安全设计；指标是合规覆盖率，不是激活/留存 |
| 内部工具 / 纯技术基础设施 | "用户"是内部员工，需求相对明确，不需市场验证 | 需求收集（内部访谈+观察）→ 快速原型 → 内测迭代；看任务耗时/错误率/满意度 |
| 已成熟业务做精细化运营 | PMF 早达成、飞轮在转，问题是把每层漏斗做更高效 | 各层指标精细化（A/B、漏斗分析、分群运营）+ OKR |
| 创作/内容类（重主观品质） | 用量化指标衡量电影/游戏是否 PMF 会导向平庸 | 定性研究 + 同行评审 + 核心粉丝深度反馈；数据是辅助不是主导 |

判断要点：**命中不适用场景时，不要给该项目套 Demo/MVP/PMF 阶段或 AARRR 指标**，而是直接说明它属于哪类、应该换哪套思路。
